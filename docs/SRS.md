## 1. Introduction

### 1.1 Background

HomeBite is a web and mobile platform that enables families to send homemade food to loved ones through a network of verified delivery partners. It is a **home-to-kin logistics service**, not a restaurant marketplace.

The core transaction: a Sender (e.g., a mother) schedules a daily tiffin pickup from her home. A verified Rider collects the packed food and delivers it to the Sender's registered Receiver (e.g., her son at a hostel). The Receiver confirms receipt with a one-time OTP.

HomeBite occupies a distinct gap in the Indian food-logistics market. No existing platform — Zomato, Swiggy, Dunzo, or Porter — is designed or optimised for this specific use case: daily, scheduled, home-origin, family-food delivery with OTP-based confirmation and subscription billing.

### 1.2 System Overview

The HomeBite system comprises the following logical components:

| Component | Description | Primary Users |
|---|---|---|
| **Sender Web App** | Responsive web app for scheduling, payment, and tracking | Sender |
| **Receiver Web App** | Lightweight app for tracking and OTP confirmation | Receiver |
| **Rider Mobile App** | PWA / native app for job management and GPS | Delivery Partner |
| **Admin Dashboard** | Web-based ops and management panel | Admin, Super Admin |
| **Backend API** | RESTful Node.js API (Next.js App Router) | All clients |
| **Job Scheduler** | Nightly cron process for DeliveryJob generation | System |
| **Notification Service** | FCM push + SMS gateway | All users |
| **Real-Time Layer** | WebSocket / Socket.io for live GPS tracking | Sender, Receiver, Rider |

### 5.1 Authentication and Identity Management

#### 5.1.1 User Registration

**SRS-FR-001** [P1] — Phone-Based Registration  
The system shall allow any new user to register using their mobile phone number. Registration requires:  
- A valid 10-digit Indian mobile number  
- OTP verification (6-digit, sent via SMS)  
- Role selection during first registration (Sender / Receiver / Rider)  
- Full name (minimum 2 characters, maximum 60 characters)  

*Traceability:* PRD-FR-01 | *Test:* Register with valid number, invalid number, duplicate number

**SRS-FR-002** [P1] — OTP Generation and Delivery  
The system shall generate a cryptographically random 6-digit OTP for every registration and login request. The OTP must:  
- Be delivered via SMS within 30 seconds of request  
- Expire after 10 minutes or first successful use, whichever is earlier  
- Be rate-limited to a maximum of 3 OTP requests per phone number per hour  
- Be invalidated on each new request (previous OTPs cannot be reused)  

*Traceability:* PRD-FR-01 | *Test:* Verify expiry, verify rate limit, verify invalidation on re-request

**SRS-FR-003** [P1] — Role-Based Post-Login Routing  
Upon successful login, the system shall redirect the user to their role-specific dashboard:  
- Sender → `/dashboard/sender`  
- Receiver → `/dashboard/receiver`  
- Rider → `/dashboard/rider`  
- Admin → `/dashboard/admin`  
- Super Admin → `/dashboard/superadmin`  

A user registered under multiple roles shall be prompted to select an active session role.  

*Traceability:* PRD-FR-04 | *Test:* Login as each role, verify redirect destination

**SRS-FR-004** [P1] — JWT Session Management  
The system shall issue a JWT access token (15-minute TTL) and a refresh token (30-day TTL) upon successful authentication. The system shall:  
- Silently refresh the access token using the refresh token before expiry  
- Invalidate all tokens on explicit logout  
- Invalidate refresh tokens if the user account is suspended  
- Store refresh tokens server-side in a token blacklist-capable store (Redis)  

*Traceability:* PRD-NFR-10 | *Test:* Token expiry handling, logout invalidation, suspended account

**SRS-FR-005** [P1] — Profile Management  
All authenticated users shall be able to update their profile. Editable fields:  
- Full name  
- Profile photo (JPEG/PNG, max 2MB, stored in CDN)  
- For Senders: home pickup address  
- For Riders: vehicle type, vehicle registration number  

Phone number is not editable post-registration without identity re-verification.  

*Traceability:* PRD-FR-05 | *Test:* Update each editable field, attempt phone number change

### 5.2 Sender — Receiver Profile Management

**SRS-FR-006** [P1] — Add Receiver Profile  
A Sender shall be able to add up to 5 Receiver profiles. Each profile must capture:  
- Receiver's full name (required)  
- Relationship label: Son, Daughter, Husband, Wife, Friend, Other (required)  
- Receiver's mobile phone number (required; used for SMS notifications)  
- Primary drop-off address (full address with pincode) (required)  
- Landmark / building notes (optional, max 200 characters)  
- Delivery gate / floor / flat number (optional, max 100 characters)  

*Traceability:* PRD-FR-06, PRD-FR-07 | *Test:* Add 5 receivers, attempt to add 6th (must fail)

**SRS-FR-007** [P1] — Link Existing User as Receiver  
If the phone number entered during receiver creation matches an existing HomeBite user account, the system shall:  
- Display the matching user's name for confirmation  
- On confirmation, link the two accounts so the Receiver's dashboard reflects incoming deliveries from the Sender  
- Not create a duplicate account  

*Traceability:* PRD-FR-09 | *Test:* Link existing user, link non-existing user (no match scenario)

**SRS-FR-008** [P1] — Manage Receiver Profiles  
A Sender shall be able to:  
- Edit any field of a receiver profile  
- Pause a receiver profile (suspends all active subscriptions for that receiver without deletion)  
- Delete a receiver profile (allowed only if no active or pending DeliveryJobs exist for that profile; otherwise, request deferred to next cycle)  

*Traceability:* PRD-FR-08 | *Test:* Edit, pause, delete with active job (must be blocked), delete without active job

### 5.3 Subscription Management

**SRS-FR-009** [P1] — Create Subscription  
A Sender shall be able to create a subscription for a receiver profile. A subscription must capture:  
- Associated receiver profile (required)  
- Subscription plan tier: Single Trip | 7-Day Pass | Monthly Plan (required)  
- Pickup time window (required): selectable 30-minute slots between 09:00 AM and 02:00 PM  
- Pickup address: defaulted from Sender profile; editable per subscription  
- Active days: weekdays only (Mon–Fri) by default; configurable to include Saturday  

A Sender may not create more than one active subscription per receiver profile.  

*Traceability:* PRD-FR-10, PRD-FR-11, PRD-FR-12 | *Test:* Create each plan type, attempt duplicate subscription

**SRS-FR-010** [P1] — Subscription Plan Specifications  

| Plan | Delivery Days | Billing | Cancellation |
|---|---|---|---|
| Single Trip | 1 day only | ₹59 flat | No refund once job created |
| 7-Day Pass | 6 weekdays (Mon–Sat configurable) | ₹299 flat | Pro-rated wallet credit |
| Monthly Plan | 26 weekdays | ₹999/month | Per BR-03 |

The system shall store the plan type, start date, end date, billing amount, and payment reference for every subscription.  

*Traceability:* PRD-Section 11 (MVP), PRD-BR-03 | *Test:* Verify each plan's day count, billing amount, cancellation outcome

**SRS-FR-011** [P1] — Pause Subscription  
A Sender shall be able to pause an active subscription for a date range. Rules:  
- Pause must be requested at least 2 hours before the pickup window of the earliest affected day  
- A pause range may not exceed 14 consecutive calendar days  
- Multiple pause windows per subscription are allowed, but they may not overlap  
- Paused days do not generate DeliveryJobs  
- Paused days are credited to the subscription end date (rolled forward) rather than refunded  

*Traceability:* PRD-FR-13, PRD-BR-02 | *Test:* Pause within 2-hr window (must fail), pause valid range, overlapping pause (must fail)

**SRS-FR-012** [P1] — Cancel Subscription  
A Sender may cancel an active subscription at any time. On cancellation:  
- If the cancellation is before the 15th calendar day of the subscription: 50% refund of unused day value credited to the Sender's HomeBite wallet  
- If the cancellation is on or after the 15th: unused day value credited to wallet only (no cash refund)  
- Any pending DeliveryJob for today that is not yet "Picked Up" is cancelled  
- Any job in "Picked Up" or later status continues to completion  

*Traceability:* PRD-BR-03 | *Test:* Cancel before day 15, cancel after day 15, cancel with in-transit job

**SRS-FR-013** [P2] — Subscription Renewal  
The system shall notify the Sender 3 days before a Monthly Plan expires. The Sender can:  
- Renew with the same plan (one-click, last payment method)  
- Upgrade to a different plan  
- Let the subscription expire (no auto-renewal without explicit action)  

*Traceability:* PRD-FR-14 | *Test:* Renewal notification trigger, one-click renewal, expiry without renewal

### 5.4 Payment Processing

**SRS-FR-014** [P1] — Payment Initiation  
When a Sender confirms a subscription, the system shall:  
1. Call the Razorpay Orders API (`POST /v1/orders`) on the backend with the plan amount, currency (INR), and a unique receipt ID  
2. Return the Razorpay `order_id` to the frontend  
3. Open the Razorpay checkout modal client-side with the `order_id`  
4. Support payment via: UPI, debit card, credit card, net banking, Paytm wallet  

*Traceability:* PRD-FR-16, PRD-FR-17 | *Test:* Successful payment flow, cancelled payment, network timeout during payment

**SRS-FR-015** [P1] — Payment Verification  
Upon Razorpay returning payment success to the client, the system shall:  
1. Receive `razorpay_payment_id`, `razorpay_order_id`, and `razorpay_signature` from the client  
2. Verify the signature server-side using HMAC-SHA256: `HMAC_SHA256(order_id + "|" + payment_id, razorpay_key_secret)`  
3. Only activate the subscription if signature verification succeeds  
4. Reject and log any request where signature verification fails, without activating the subscription  

*Traceability:* PRD-FR-17, PRD-BR-01 | *Test:* Valid signature → subscription activated, tampered signature → rejected

**SRS-FR-016** [P1] — Payment Receipt  
On successful payment activation, the system shall:  
- Store a Payment record (amount, plan, timestamp, Razorpay payment ID, subscription ID)  
- Send an in-app notification to the Sender with a receipt summary  
- Send an SMS receipt to the Sender's registered phone number  

*Traceability:* PRD-FR-18 | *Test:* Verify receipt stored, notification sent, SMS delivered

**SRS-FR-017** [P1] — Failed Payment Handling  
If a payment fails (declined, timeout, or user abort):  
- The subscription remains in `PENDING_PAYMENT` status  
- The system presents the Sender with a retry option  
- After 3 failed attempts within 24 hours, the Sender receives an SMS prompting manual retry  
- The subscription is automatically cancelled after 24 hours in `PENDING_PAYMENT` status  

*Traceability:* PRD-FR-19 | *Test:* Simulate payment failure, verify retry, verify auto-cancellation at 24h

**SRS-FR-018** [P2] — Admin-Initiated Refunds  
An Admin shall be able to initiate a full or partial refund against any completed Razorpay payment by:  
1. Entering a refund amount (must not exceed original payment amount)  
2. Selecting a reason code (Delivery Failed / Rider No-Show / System Error / Goodwill)  
3. Submitting to Razorpay Refunds API (`POST /v1/payments/{id}/refund`)  
4. Recording the refund reference and reason in the payment record  

*Traceability:* PRD-FR-20 | *Test:* Full refund, partial refund, refund exceeding original (must fail)

### 5.5 Job Generation (System — Backend)

**SRS-FR-019** [P1] — Nightly Job Generation Cron  
A scheduled process shall run every day at 23:30 IST (UTC+5:30). This process shall:  
1. Query all Subscription documents where `status = ACTIVE` and the next calendar day falls within the subscription's active period and is not in a pause window  
2. For each qualifying subscription, create one DeliveryJob document  
3. Assign a cryptographically random 4-digit OTP to each job  
4. Attempt to assign each job to an available, active Rider in the same delivery zone  
5. Send the assigned Rider a push notification summarising tomorrow's route  
6. Log the total jobs created, assigned, and unassigned with a timestamp  

The cron process must complete within 5 minutes for up to 10,000 active subscriptions.  

*Traceability:* PRD-FR-21, PRD-FR-22, PRD-FR-23, PRD-FR-24 | *Test:* Run cron with 0, 100, 1,000, 10,000 subscriptions; verify paused subscriptions skipped

**SRS-FR-020** [P1] — DeliveryJob States  
Each DeliveryJob shall follow this strict state machine:

```
PENDING_ASSIGNMENT
      │
      ▼
  ASSIGNED ──────────────────────────────┐
      │                                   │
      ▼                                   │
  PICKED_UP ── (rider flags issue) ─→ ATTEMPTED_FAILED
      │
      ▼
  IN_TRANSIT
      │
      ▼
  DELIVERED
```

State transitions shall only occur via explicit API calls with the correct authenticated role. No state may be skipped.  

*Traceability:* PRD (Flow 3) | *Test:* Valid transitions for each state, attempt invalid transition (must be rejected)

**SRS-FR-021** [P1] — Unassigned Job Escalation  
If a DeliveryJob remains in `PENDING_ASSIGNMENT` status at T-2 hours before the subscription's pickup window start time, the system shall:  
- Send a push notification and email to all Admin users  
- Display the job in a dedicated "Needs Attention" panel in the Admin dashboard  
- Allow Admin to manually assign the job to any available Rider  

*Traceability:* PRD-FR-25 | *Test:* Create unassigned job, advance time to T-2hr, verify alert sent

**SRS-FR-022** [P1] — OTP Management  
OTPs shall be:  
- Generated as a 4-digit integer (0000–9999) using a cryptographically secure random function  
- Stored hashed (bcrypt, cost factor 10) in the DeliveryJob document  
- Visible only to the Receiver on their authenticated delivery screen (never to the Sender or Rider)  
- Valid only until the job reaches `DELIVERED` or `ATTEMPTED_FAILED` status, or for a maximum of 4 hours after the pickup window, whichever comes first  
- Rate-limited: maximum 5 incorrect entry attempts; job auto-escalated to Admin on 5th failure  

*Traceability:* PRD-FR-23, PRD-NFR-13, PRD-BR-06, PRD-BR-08 | *Test:* Valid OTP → deliver, incorrect OTP × 5 → escalate, expired OTP → reject

### 5.6 Rider Application

**SRS-FR-023** [P1] — Rider Job List  
An authenticated Rider shall see a dashboard showing all DeliveryJobs assigned to them for the current day. The list shall:  
- Be sorted by geographic pickup cluster (nearest pickup first within the same time window)  
- Display per job: Sender name, pickup address, pickup window, Receiver name, drop-off address  
- Distinguish between upcoming, in-progress, and completed jobs visually  
- Refresh automatically every 60 seconds without a page reload  

*Traceability:* PRD-FR-26, PRD-FR-27 | *Test:* Rider with 0, 1, 5, 15 jobs; verify sort order; verify auto-refresh

**SRS-FR-024** [P1] — Mark Picked Up  
When a Rider arrives at the pickup address and receives the tiffin, they shall tap "Mark Picked Up". This action shall:  
1. Transition the DeliveryJob from `ASSIGNED` to `PICKED_UP` and then immediately to `IN_TRANSIT`  
2. Record the pickup timestamp and the Rider's GPS coordinates at the moment of action  
3. Send a push notification to the Sender: "Rider [Name] has picked up your tiffin. Estimated arrival: [ETA]."  
4. Send a push notification to the Receiver: "Your tiffin is on the way! Track the rider."  
5. Begin broadcasting the Rider's GPS location to Sender and Receiver via WebSocket  

*Traceability:* PRD-FR-28 | *Test:* Tap picked up, verify status transitions, verify notifications sent, verify GPS broadcast begins

**SRS-FR-025** [P1] — OTP Entry and Delivery Confirmation  
When the Rider reaches the Receiver, they shall request the OTP and enter it. The system shall:  
1. Validate the entered OTP against the stored hash  
2. On match: transition job to `DELIVERED`, record delivery timestamp and Rider GPS, stop GPS broadcast, credit Rider earnings, notify Sender and Receiver  
3. On mismatch: increment OTP attempt counter, display remaining attempts to Rider  
4. On 5th mismatch: lock the OTP field, flag job for Admin review, notify Admin  

*Traceability:* PRD-FR-29, PRD-BR-06, PRD-BR-08 | *Test:* Correct OTP, 1–4 incorrect attempts, 5th incorrect → lock

**SRS-FR-026** [P1] — Flag Failed Delivery  
A Rider shall be able to flag a job as `ATTEMPTED_FAILED`. Required fields:  
- Reason code (mandatory): RECEIVER_NOT_PRESENT | WRONG_ADDRESS | GATE_BLOCKED | OTHER  
- Free-text note (optional, max 200 characters)  
- The system records the Rider's GPS location and timestamp at the time of flagging  

This action shall trigger the Dispute flow (Section 5.9).  

*Traceability:* PRD-FR-30, PRD-D-06 | *Test:* Flag with each reason code, verify GPS recorded, verify dispute triggered

**SRS-FR-027** [P1] — Rider Earnings Dashboard  
A Rider shall see a real-time earnings view showing:  
- Today's earnings (sum of all completed jobs × per-delivery rate)  
- Pending wallet balance (earnings not yet paid out)  
- Cumulative monthly earnings  
- Per-job breakdown: job ID, delivery time, amount, status  
- Payout history with dates and amounts  

Earnings shall update within 60 seconds of a job reaching `DELIVERED` status.  

*Traceability:* PRD-FR-31 | *Test:* Complete a delivery, verify earnings updated, verify breakdown accuracy

**SRS-FR-028** [P2] — Rider Payout Request  
A Rider shall be able to request a payout of their pending wallet balance once per calendar day after 18:00 IST. The system shall:  
- Submit the payout to the Rider's registered bank account via Razorpay Fund Transfer API  
- Mark the payout as `PROCESSING` until Razorpay confirms  
- Update wallet balance to zero on successful transfer  
- Notify the Rider via push and SMS on success or failure  

*Traceability:* PRD-FR-32, PRD-BR-12 | *Test:* Request payout before 18:00 (must fail), after 18:00 (must succeed), verify wallet zeroed

### 5.7 Live GPS Tracking

**SRS-FR-029** [P1] — GPS Location Broadcasting  
When a DeliveryJob is in `IN_TRANSIT` status, the Rider's mobile app shall transmit their GPS coordinates to the backend every 10 seconds. The system shall:  
- Accept coordinates via a dedicated WebSocket channel authenticated with the Rider's JWT  
- Broadcast the latest coordinates to all authenticated Sender and Receiver sessions subscribed to that specific DeliveryJob  
- Store each GPS point in a time-series log (for dispute resolution; retained 30 days)

## 9. Business Rules

> Business rules define system-enforced logic that cannot be overridden by any user role unless explicitly stated.

### 9.1 Subscription and Billing Rules

**SRS-BR-001** [P1] — Subscription Activation Gate  
A subscription shall transition to `ACTIVE` status only after `razorpay_signature` has been verified server-side. No subscription shall be activated on the basis of client-side payment confirmation alone.

**SRS-BR-002** [P1] — Pause Credit Policy  
Paused delivery days shall be credited to the subscription's end date (i.e., the subscription is extended, not shortened). No monetary refund is issued for paused days under any circumstance.

**SRS-BR-003** [P1] — Cancellation Refund Tiers  
- Cancellation before day 15 of subscription: 50% of the monetary value of unused days credited to the HomeBite wallet  
- Cancellation on or after day 15: 100% of remaining value credited to HomeBite wallet only (no cash refund)  
- HomeBite wallet credits are non-transferable and valid for 12 months  

**SRS-BR-004** [P1] — Single Subscription Per Receiver  
A Sender may not hold more than one active or pending subscription per receiver profile at any time. Attempting to create a duplicate subscription shall return an error: "An active subscription already exists for [Receiver Name]."

**SRS-BR-005** [P1] — Receiver Profile Prerequisite  
A subscription creation attempt without a saved and verified receiver profile shall be rejected. The system shall redirect the Sender to create a receiver profile first.

### 9.2 Delivery and OTP Rules

**SRS-BR-006** [P1] — OTP-Only Delivery Confirmation  
No DeliveryJob may be transitioned to `DELIVERED` status without correct OTP entry. This rule cannot be overridden by Admin, Super Admin, or any API call. The only exception is a manual database correction by the Super Admin following a documented dispute resolution decision.

**SRS-BR-007** [P1] — OTP Visibility Restriction  
The OTP for a DeliveryJob shall only be visible to the authenticated Receiver of that specific job. It shall not appear in:  
- Sender's dashboard or notifications  
- Rider's app at any point  
- Admin's dispute panel (Admin sees the attempt count, not the OTP itself)  
- API responses to any role other than the Receiver  

**SRS-BR-008** [P1] — OTP Lock on Repeated Failure  
After 5 consecutive incorrect OTP entries for a single DeliveryJob, the OTP input shall be locked. The Rider must contact Admin support. Only an Admin can reset the OTP lock (generating a new OTP and notifying the Receiver).

**SRS-BR-009** [P1] — Pickup Window Escalation  
A DeliveryJob that has not been transitioned to `PICKED_UP` within 30 minutes of the end of the pickup window is auto-escalated to Admin. The escalation does not automatically cancel the job.

**SRS-BR-010** [P1] — Maximum Jobs Per Rider Per Day  
The job assignment algorithm and manual assignment UI shall enforce a hard cap of 15 DeliveryJobs per Rider per calendar day. Any assignment attempt exceeding this cap shall be rejected with the message: "This rider has reached the maximum delivery limit for today."

### 9.3 Rider Rules

**SRS-BR-011** [P1] — Verification Before Assignment  
A Rider account with status other than `ACTIVE` shall not receive job assignments — neither via the automatic cron assignment nor via manual Admin assignment. The system shall exclude non-active Riders from all assignment logic.

**SRS-BR-012** [P1] — Payout Timing Restriction  
Rider payout requests submitted before 18:00 IST shall be rejected with the message: "Payouts are processed after 6:00 PM. Please try again later." This rule exists to allow the payout batch to include all same-day completed deliveries.

**SRS-BR-013** [P2] — Rider No-Show Review Policy  
If a Rider has 3 or more DeliveryJobs auto-escalated as no-show (i.e., `PICKED_UP` not set within 30 minutes of pickup window end, traced to the same Rider) within any 30-day rolling period, the system shall automatically set the Rider's status to `UNDER_REVIEW` and notify Admin.

**SRS-BR-014** [P1] — Rider Phone Number Masking  
The Sender's full phone number shall not be visible to the Rider. The system shall display a masked version (e.g., 98765 ••••• ) unless the Rider's active job for that Sender has status `ASSIGNED` or `IN_TRANSIT`. After delivery or failure, the number is masked again.

### 9.4 Dispute Rules

**SRS-BR-015** [P1] — Dispute Submission Window  
A Sender may raise a dispute for a DeliveryJob only within 24 hours of the job's scheduled pickup window end time. Dispute submission attempts after 24 hours shall be rejected by the system.

**SRS-BR-016** [P1] — GPS-Based Dispute Classification  
The Admin dispute panel shall classify disputes automatically based on GPS log data:  
- If GPS log shows Rider within 200 metres of drop-off address during delivery window: classified as `RECEIVER_ABSENT`  
- If GPS log shows Rider never within 500 metres of drop-off address: classified as `RIDER_NO_SHOW`  
- If GPS data is absent or inconclusive: classified as `INDETERMINATE` (Admin manual review required)  

**SRS-BR-017** [P1] — Dispute Resolution Authority  
Admin has final resolution authority. Resolution decisions (refund, no-refund, reschedule) are irreversible once confirmed in the dispute panel. Super Admin may reverse a resolution within 48 hours of the original decision.

### 9.5 Food Safety Policy Rules

**SRS-BR-018** [P1] — No Food Quality Liability  
HomeBite is a logistics platform. The system shall not collect, store, or display food content descriptions, nutritional information, or quality claims. All liability for food safety rests with the Sender. This constraint shall be communicated in the Sender's Terms of Service.

**SRS-BR-019** [P1] — Container Integrity Policy  
Riders shall be contractually prohibited from opening tiffin containers. The system shall include this as a mandatory acknowledgement during Rider onboarding. Violation is grounds for immediate deactivation.

## 10. Security Requirements

### 10.1 Authentication Security

**SRS-SEC-001** [P1] — Transport Security  
All API endpoints, WebSocket connections, and static asset delivery shall use TLS 1.3 minimum. TLS 1.0 and 1.1 shall be disabled at the load balancer. HTTP requests shall be redirected to HTTPS with a 301 permanent redirect.

**SRS-SEC-002** [P1] — JWT Security  
- Access tokens: HS256 signed, 15-minute TTL, audience claim scoped to `homebite-api`  
- Refresh tokens: 256-bit random value (not JWT), stored server-side in Redis with 30-day TTL  
- Tokens are invalidated server-side on logout, role change, or account suspension  
- Token payload shall not contain PII (no phone number, name, or address)  

**SRS-SEC-003** [P1] — OTP Security  
- Registration / login OTPs: 6-digit, cryptographically random, 10-minute TTL  
- Delivery OTPs: 4-digit, cryptographically random, stored as bcrypt hash (cost 10)  
- OTP generation rate-limited: 3 requests per phone per hour (registration); 1 per job (delivery)  
- No OTP shall be logged in application logs or error traces  

**SRS-SEC-004** [P1] — RBAC Enforcement  
Role-based access control shall be enforced at the API middleware layer, not only in the UI. Every API route shall declare its allowed roles. Requests from roles not in the allowed set shall return HTTP 403. RBAC shall be tested with automated role-boundary tests in CI/CD.

### 10.2 Data Security

**SRS-SEC-005** [P1] — Encryption at Rest  
All data stored in MongoDB Atlas shall use encryption at rest via MongoDB's native AES-256 encryption. Encryption keys shall be managed via a cloud KMS (AWS KMS or GCP Cloud KMS).

**SRS-SEC-006** [P1] — PII Handling  
Phone numbers shall be stored in a reversibly encrypted form (AES-256 with application-level key) rather than plaintext. Only the Auth service and Notification service may decrypt phone numbers. GPS coordinates associated with a delivered or failed job shall be purged after 30 days.

**SRS-SEC-007** [P1] — Payment Data  
No raw payment card data, CVV, or UPI PIN shall be transmitted to or stored by HomeBite servers. All payment data is handled exclusively by Razorpay. HomeBite stores only: Razorpay `order_id`, `payment_id`, payment status, and amount.

**SRS-SEC-008** [P1] — KYC Document Storage  
Rider KYC documents (Aadhaar, PAN images) shall not be stored in HomeBite's primary database or CDN. They shall be transmitted directly to a compliant KYC vault (e.g., Digio, Signzy, or DigiLocker API) and only a verification token stored in the Rider record.

### 10.3 Application Security

**SRS-SEC-009** [P1] — Input Validation  
All API inputs shall be validated server-side using a schema validation library (e.g., Zod). Validation failures shall return HTTP 400 with a generic error message (no stack traces or internal detail exposed). Client-side validation is a UX convenience only and shall never be the sole validation layer.

**SRS-SEC-010** [P1] — Rate Limiting  
API rate limits shall be enforced at the gateway level (per IP and per authenticated user):  
- Authentication endpoints: 10 requests / minute / IP  
- OTP request endpoints: 3 requests / hour / phone number  
- Standard API endpoints: 100 requests / minute / authenticated user  
- Admin endpoints: 200 requests / minute / authenticated admin  
Exceeded rate limits return HTTP 429 with `Retry-After` header.

**SRS-SEC-011** [P1] — Webhook Signature Verification  
All incoming Razorpay webhooks shall be verified using HMAC-SHA256 with the Razorpay webhook secret. Webhooks with invalid or missing signatures shall return HTTP 400 and be logged (without processing). No subscription or payment state change shall occur based on an unverified webhook.

**SRS-SEC-012** [P2] — Security Headers  
All HTTP responses shall include:  
- `Strict-Transport-Security: max-age=31536000; includeSubDomains`  
- `Content-Security-Policy` (restrictive policy; inline scripts disallowed)  
- `X-Frame-Options: DENY`  
- `X-Content-Type-Options: nosniff`  
- `Referrer-Policy: no-referrer-when-downgrade`  

**SRS-SEC-013** [P2] — Audit Logging  
All write operations on sensitive entities (subscriptions, payments, rider status changes, admin dispute resolutions) shall be logged to an append-only audit log with: timestamp, user ID, role, action performed, entity affected, and previous/new values. Audit logs are retained for 2 years.

**SRS-PERF-008** [P1] — Database Query Performance

## 12. External Integrations

### 12.1 Razorpay — Payment Gateway

**SRS-INT-001** [P1] — Order Creation  
The backend shall call `POST https://api.razorpay.com/v1/orders` when a Sender confirms a subscription plan. Request parameters: `amount` (in paise), `currency: "INR"`, `receipt` (unique subscription ID), `notes` (plan type, sender ID). Response: `order_id` returned to client.

**SRS-INT-002** [P1] — Payment Verification  
On client-side payment completion, the backend shall verify the signature using: `HMAC_SHA256(razorpay_order_id + "|" + razorpay_payment_id, RAZORPAY_KEY_SECRET)`. Only a matching signature activates the subscription.

**SRS-INT-003** [P1] — Refund Initiation  
Admin-initiated refunds shall call `POST https://api.razorpay.com/v1/payments/{payment_id}/refund` with `amount` (in paise, partial allowed) and `notes.reason`. The response `id` is stored as the refund reference.

**SRS-INT-004** [P2] — Fund Transfer (Rider Payouts)  
Rider payouts shall use Razorpay Route (Fund Transfers API) or Razorpay X Payouts. Each payout call: `POST /v1/fund_transfers` with linked Rider bank account `account_id`, amount, and narration "HomeBite Rider Payout — [Date]". Response `id` stored in Rider payout log.

**Environment:** Razorpay Test environment for dev/staging; Production keys in environment variables only, never in source code.

### 12.2 Google Maps Platform

**SRS-INT-005** [P1] — Geocoding API  
When a Sender or Rider enters a pickup or drop-off address, the system shall call `GET https://maps.googleapis.com/maps/api/geocode/json?address={address}&key={API_KEY}` to validate the address and extract `lat/lng` coordinates. Invalid addresses (zero results) shall prompt the user to refine the input.

**SRS-INT-006** [P1] — Maps JavaScript API (Tracking View)  
The live tracking screen shall embed a Google Maps instance using the Maps JavaScript API. The map shall display:  
- A custom rider marker (motorcycle icon) at the last known GPS coordinate  
- A destination pin at the drop-off address  
- The marker shall animate smoothly between GPS updates (interpolated movement)  

**SRS-INT-007** [P1] — Directions API (ETA Calculation)  
To display ETA on the tracking view, the system shall call `GET https://maps.googleapis.com/maps/api/directions/json` with the Rider's current `lat/lng` as origin and the drop-off address as destination, mode `driving`. ETA is recalculated every 60 seconds. The result is cached client-side between calls.

**Usage Controls:** API key restricted by HTTP referrer (frontend) and IP (backend). Quotas monitored via Google Cloud Console; budget alert set at 80% of monthly quota.

---

### 12.3 Firebase Cloud Messaging (FCM)

**SRS-INT-008** [P1] — Push Notification Delivery  
The backend shall use the FCM HTTP v1 API (`POST https://fcm.googleapis.com/v1/projects/{project_id}/messages:send`) to deliver push notifications. Each notification includes: `title`, `body`, `data` payload (job ID, action type for deep linking), and `notification` block. Tokens are stored per-user per-device (a user may have multiple devices).

**SRS-INT-009** [P1] — FCM Token Management  
FCM device tokens shall be refreshed and stored by the client on each app launch. Stale tokens (returned as `UNREGISTERED` by FCM) shall be removed from the database immediately upon FCM error response. A user with no valid FCM tokens falls back to SMS only.

---

### 12.4 SMS Gateway (MSG91 / Twilio)

**SRS-INT-010** [P1] — SMS Delivery  
SMS shall be sent via MSG91 (primary, India-optimised) or Twilio (fallback). All messages must use DLT-registered templates. API integration: `POST https://api.msg91.com/api/v5/flow/` with `template_id`, `mobile`, and variable substitutions. Delivery reports shall be fetched via webhook callback.

**SRS-INT-011** [P1] — DLT Template Compliance  
All SMS content is fixed-format with variable placeholders. Templates must be registered with the TRAI DLT portal before go-live. No transactional SMS may be sent using an unregistered template. A library of approved templates shall be maintained in an environment configuration file.

**Required DLT Templates (minimum):**

| Template ID | Use Case |
|---|---|
| HB-T-001 | OTP for registration / login |
| HB-T-002 | Rider picked up tiffin (Sender) |
| HB-T-003 | Tiffin delivered (Sender) |
| HB-T-004 | Tiffin on the way (Receiver) |
| HB-T-005 | Delivery failed (Sender and Receiver) |
| HB-T-006 | Subscription expiry reminder (Sender) |
| HB-T-007 | Payment receipt (Sender) |

### 12.5 MongoDB Atlas

**SRS-INT-012** [P1] — Database Connection  
The backend shall connect to MongoDB Atlas using the official Node.js driver (mongoose). Connection pooling: minimum 5, maximum 50 connections per API server instance. Connection string stored as environment variable; never in source code.

**SRS-INT-013** [P1] — Cluster Configuration  
MongoDB Atlas cluster shall be deployed in region `ap-south-1` (Mumbai). Minimum cluster tier: M10 for pilot launch; upgrade path to M30 at 50,000 daily jobs. Automated daily backups with 7-day retention. Point-in-time recovery enabled.

---

### 12.6 Redis (BullMQ Job Queue)

**SRS-INT-014** [P1] — Queue Infrastructure  
A Redis instance (AWS ElastiCache or Redis Cloud) shall underpin:  
- BullMQ queues for: notification delivery, payout processing, cron job execution  
- JWT refresh token store  
- OTP rate-limit counters  
- GPS coordinate caching (latest position per active job)  

Redis shall be deployed in the same region as the API servers (`ap-south-1`). Persistence: RDB snapshots every 15 minutes (not AOF, to reduce latency).

## 14. Error Handling and Edge Cases

### 14.1 Critical Edge Cases

| Scenario | System Behaviour |
|---|---|
| Rider GPS signal lost mid-transit | Last known position displayed; tracking shows "Location updating..." banner; GPS trail gap logged |
| Cron job fails at midnight | BullMQ retry with exponential backoff (3 retries); PagerDuty alert on final failure; manual trigger available via Admin panel |
| Razorpay webhook not received after payment | Reconciliation job runs every 30 minutes; queries Razorpay for orders in PENDING state older than 15 minutes |
| Receiver does not rate within 2 hours | Rating prompt auto-dismisses; delivery still marked complete; no rating recorded |
| Sender pauses subscription after cron generates today's job | Today's job proceeds; pause applies from the following day |
| Rider marks picked up but then loses device | Admin can manually close the job after GPS evidence and phone confirmation; OTP rule still applies unless Super Admin overrides |
| Two riders attempt to mark same job as picked up (race condition) | Atomic MongoDB `findOneAndUpdate` with status check; only the first successful update prevails; second rider receives error |
| Receiver uninstalls app and has no FCM token | SMS fallback sends OTP separately to Receiver's phone; OTP still required for delivery |
| Network failure during OTP submission | Client retries submission up to 3 times; idempotency key prevents duplicate state transitions |

### 14.2 Error Response Standards

All API errors shall follow this response schema:
```json
{
  "success": false,
  "error": {
    "code": "SUBSCRIPTION_DUPLICATE",
    "message": "An active subscription already exists for this receiver.",
    "field": "receiverProfileId"
  }
}
```

HTTP status codes:
- `400` Bad Request — validation failure or business rule violation
- `401` Unauthorized — missing or invalid token
- `403` Forbidden — authenticated but insufficient role
- `404` Not Found — resource does not exist
- `409` Conflict — state conflict (duplicate, race condition)
- `422` Unprocessable Entity — logically invalid request
- `429` Too Many Requests — rate limit exceeded
- `500` Internal Server Error — unexpected failure (generic message only; details in logs)

No stack traces, internal messages, or database error strings shall be exposed in API error responses.
